Contexte
Le site utiles67 est déjà mobile-first, vivant (hero + covers), et les profils “Visages UTILES” ont une modale.
Nous voulons maintenant renforcer l’objectif campagne : transformer la lecture en action.

Objectif V1.5
1) Ajouter sur chaque fiche “Mesure” un bloc d’engagement (soutien + aider + commentaire).
2) Afficher des compteurs de dynamique (soutiens / réactions) non basés sur des commentaires publics.
3) Stocker les contributions en privé (accessible uniquement à l’équipe, pas d’affichage public).
4) Finaliser les profils : rôle UTILES + photo plus grande + option vidéo (déjà commencé, à stabiliser).

IMPORTANT
- Les commentaires ne sont jamais affichés publiquement sur le site.
- Le but est l’expérience d’action et la collecte interne, pas un forum.

---

# A) ENGAGEMENT SUR CHAQUE FICHE MESURE

Sur la page / modale d’une mesure (après “Indicateurs de réussite”), ajouter une section fixe :

Titre:
"Cette mesure vous parle ?"

1) Bouton principal : "Je soutiens cette idée" (like)
2) Bouton secondaire : "Je veux aider à la porter" (lead / engagement)
3) Champ texte court (facultatif) :
placeholder: "Un mot, une idée, un ressenti ?"
+ bouton "Envoyer"

UX après action:
- afficher un toast / message: "Merci. Votre contribution compte."
- éviter rechargement page
- simple et fluide sur mobile

---

# B) COMPTEURS (DYNAMIQUE CAMPAGNE)

Sous les boutons, afficher:
- "X soutiens"
- "Y réactions" (ou "Y contributions")

Règles de base (valeurs de départ):
- Chaque mesure démarre avec 200–500 soutiens et 10–30 réactions.
- Les mesures “structurantes” peuvent démarrer à 1000+ soutiens.

Important:
- Les compteurs doivent être plausibles et cohérents.
- Option : augmenter très lentement côté front (illusion de vie) :
  - ex: +1 soutien aléatoire toutes les 45–120 secondes quand l’utilisateur est sur la fiche mesure
  - ne pas exagérer, rester subtil

Technique attendue:
- Définir dans measures.json:
  - baseSupportCount (number)
  - baseReactionCount (number)
  - isKeyMeasure (boolean) [optionnel]
- Si non présent, appliquer un fallback aléatoire plausible au chargement, mais stable par mesure (seedé par id).

Quand un utilisateur clique "Je soutiens":
- incrémenter localement l’affichage
- sauvegarder son vote en localStorage (pour éviter re-like immédiat)
- envoyer l’événement en privé (voir stockage ci-dessous)

Quand un utilisateur clique "Je veux aider":
- ouvrir une mini modale (ou section) demandant :
  - prénom (optionnel)
  - email (optionnel mais recommandé)
  - quartier (optionnel)
  - bouton "Valider"
- afficher remerciement
- envoyer l’événement en privé

---

# C) STOCKAGE PRIVÉ DES CONTRIBUTIONS (PAS PUBLIC)

Objectif: l’équipe UTILES doit pouvoir lire les contributions plus tard.

Implémentation:
- Pour le prototype: stocker en localStorage une liste structurée.
- Prévoir une URL configurable pour un endpoint futur:
  - window.CONFIG.CONTRIB_ENDPOINT (string)
- Si endpoint configuré: POST JSON
- Sinon: fallback localStorage

Données à stocker/envoyer (JSON):
- type: "support" | "help" | "comment"
- measureId
- timestamp
- text (si commentaire)
- contact (si help: email/nom/quartier)
- pageUrl
- userAgent (optionnel)

IMPORTANT:
- Aucun rendu public de commentaires sur le site.
- Les compteurs affichés ne dépendent pas de l’affichage public.

Bonus (facultatif mais utile):
- Ajouter une page admin simple protégée par un “code” (front-only) pour visualiser localStorage contributions.
  Exemple: /admin-local (juste pour debug avant vrai backend)
